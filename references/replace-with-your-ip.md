# Replace the Sample Character With Your Own IP

## Goal

This skill should not lock users into the sample "Q版阿清" character. The user should replace the sample with their own repeatable IP character system.

The replacement process has three outputs:

1. An IP character card.
2. A mode system.
3. A style variant library.

## Step 1: Extract Fixed Recognition Points

Ask the user for one or more character reference images. If no image is available, ask for a written description.

Extract:

```text
Face:
Hair:
Age impression:
Body proportion:
Core outfit:
Signature accessory:
Color palette:
Personality:
Professional role:
What must never change:
What can change:
```

Keep no more than 5-7 fixed recognition points. Too many fixed traits make the character hard to adapt.

Good fixed traits:

- Black short hair
- Round eyes
- Black jacket
- Cream bear hat
- White sneakers
- Warm mentor energy

Weak fixed traits:

- "Nice"
- "High-end"
- "Cute"
- "AI feeling"

These are moods, not recognition points.

## Step 2: Define Character Modes

A strong cover system usually needs at least two modes:

| Mode | Purpose |
|---|---|
| Authority mode | For trust, teaching, methods, cases, services |
| Friendly mode | For beginner content, resources, light topics, recurring columns |

Optional modes:

| Mode | Purpose |
|---|---|
| Alert mode | For mistakes, warnings, pitfalls |
| Challenge mode | For fast-action, hot topics, experiments |
| Studio mode | For behind-the-scenes, process, design work |

Each mode should keep the same face but allow different clothing, props, and posture.

## Step 3: Define What Can Change

Separate the system into stable and flexible parts:

| Layer | Stable | Flexible |
|---|---|---|
| Face | Hair, eyes, face shape | Expression |
| Outfit | Signature silhouette | Color accents, seasonal details |
| Props | Content-related only | Computer, card, board, warning sign |
| Background | Account color logic | Topic-specific cards and stickers |
| Typography | Large readable title | Tag shape and accent color |

If a layer changes too often, the account loses recognition. If nothing changes, the covers become boring.

## Step 4: Build a Style Variant Library

For each visual variant, define:

```text
Variant name:
Best for:
Mood:
Character outfit:
Hat/accessory:
Palette:
Common props:
Background elements:
Avoid:
```

Example:

```text
Variant name: Cream Companion
Best for: beginner tutorials and soft explainers
Mood: safe, warm, low-pressure
Character outfit: cream hoodie or light utility jacket
Hat/accessory: cream bear hat
Palette: white, cream, warm yellow, cyan accents
Common props: small laptop, prompt card, sticky notes
Background elements: soft cards, arrows, checklist stickers
Avoid: too many toy elements, candy overload
```

## Step 5: Write the Character Prompt Block

Create one reusable block for each mode.

Professional mode example:

```text
Main character is [character name], a 3D chibi personal IP character:
[fixed face traits].
Wearing [signature professional outfit].
The mood is professional, clear, and trustworthy, like a creator explaining a practical workflow.
Keep the face and outfit recognizable across covers.
```

Friendly mode example:

```text
Main character is [character name], a 3D chibi personal IP character:
[fixed face traits].
Wearing [friendly outfit/accessory].
The mood is warm, approachable, and companion-like, but still connected to [creator's professional field].
Do not turn the character into a generic toy mascot.
```

## Step 6: Replace Sample Terms

When adapting this skill, replace:

| Sample term | Replace with |
|---|---|
| Q版阿清 | Your character name |
| AI+设计提效 | Your account positioning |
| AI workflow | Your content domain |
| Bear-hat friendly mode | Your friendly character mode |
| Professional mentor mode | Your authority character mode |

## Step 7: Validate Replacement

Before using the new system, generate four style-test covers with the same title and topic.

Evaluate:

1. Does the character still look like the same person?
2. Can each variant support a different content type?
3. Is at least one variant strong enough to become the main style?
4. Is the system too cute, too serious, or too generic?

Only after this test should the user use the system for real content.
